Lists & List Comprehensions

Python lists are dynamic arrays. List comprehensions provide a concise way to create lists.

# List comprehension template [expression for item in iterable if condition] # Slicing syntax: list[start:stop:step] nums = [0, 1, 2, 3, 4, 5] evens = [x for x in nums if x % 2 == 0] # [0, 2, 4] reverse = nums[::-1] # [5, 4, 3, 2, 1, 0]

ACID Properties

The four properties required to guarantee database transactions are processed reliably:

  • Atomicity: Entire transaction succeeds, or entire transaction rolls back.
  • Consistency: Database goes from one valid state to another.
  • Isolation: Transactions execute independently without interleaving.
  • Durability: Once committed, changes survive power outages or crashes.

Normalization Forms

Steps to design clean tables and eliminate anomalies:

  • 1NF: Atomic values; no repeating groups.
  • 2NF: Meets 1NF; eliminates partial dependencies (all non-key columns depend on the entire primary key).
  • 3NF: Meets 2NF; eliminates transitive dependencies (non-key columns do not depend on other non-key columns).

Core Joins

-- INNER JOIN: Matches rows in both SELECT * FROM users u INNER JOIN orders o ON u.id = o.user_id; -- LEFT JOIN: All from left, matches from right SELECT * FROM users u LEFT JOIN orders o ON u.id = o.user_id;

CAP Theorem

A distributed data store can simultaneously provide at most two of the three guarantees:

  • Consistency: Every read receives the most recent write or an error.
  • Availability: Every request receives a non-error response, without the guarantee that it contains the most recent write.
  • Partition Tolerance: System continues to operate despite an arbitrary number of messages being dropped/delayed by network partitions.

Load Balancing Strategies

How traffic is distributed across healthy application nodes:

  • Round Robin: Sequentially passes requests to servers.
  • Least Connections: Directs traffic to the server with the fewest active sessions.
  • IP Hash: Maps client's IP to a specific target server.

Caching Policies

Caching strategies for optimal read/write throughput:

  • Write-Through: Data is written to both cache and backend DB in parallel (consistency but high write latency).
  • Write-Back (Write-Behind): Written only to cache first; database update is deferred/batched (high write performance, risk of loss).
  • Cache-Aside: Application queries cache; on miss, loads from DB and updates cache.

Loss Functions

  • Mean Squared Error (Regression): $$MSE = \frac{1}{n} \sum_{i=1}^{n} (y_i - \hat{y}_i)^2$$
  • Binary Cross-Entropy (Binary Classification): $$BCE = -\frac{1}{n} \sum_{i=1}^{n} [y_i \log(\hat{y}_i) + (1-y_i) \log(1-\hat{y}_i)]$$
